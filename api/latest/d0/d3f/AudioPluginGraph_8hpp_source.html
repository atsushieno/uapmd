12 // AudioPluginGraph is supposed to abstract away how it internally routes audio plugin instances.
13 // It is supposed to `process()` audio and UMP inputs. In that sense, it is a track on a DAW alike.
14 //
15 // It can process nodes in a simple linear way, or in a perfect DAG that handles complex routing.
16 // As a minimum requirement, it must support a linear chain of nodes, by "add" and "remove" functions.
17 //
18 // Note that it is NOT to represent an entire audio graph that is supposed to involve audio device nodes.
19 // That should be handled by a sequencer, which is out of the scope of this class (not even of this library).
20 //
21 // Usually an audio plugin instance shares its lifetime with `AudioPluginNode` when it is being added to the graph,
22 // but the instance is "owned" by the `AudioPluginHostingAPI` implementation.
23 // `appendNodeSimple()` takes a delegate function called when the instance is being removed by `removeNodeSimple()`
24 // or whatever that removes `AudioPluginNode`, to help this mechanism.
